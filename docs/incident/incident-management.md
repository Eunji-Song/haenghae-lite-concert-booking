# 장애 탐지 및 대응 시나리오 보고서

## 1. 개요
본 문서는 콘서트 좌석 예약 시스템을 대상으로 수행한 부하 테스트 결과를 기반으로,
시스템의 병목 지점을 식별하고 장애를 **어떻게 탐지하고**, **어떻게 대응할 것인지**에 대한
운영 관점의 장애 대응 매뉴얼을 정리한 문서이다.

---

## 2. 시스템 전반 구조 요약
- Client → API Gateway → Application(Server)
- 주요 컴포넌트
  - 인증/인가 (JWT)
  - 대기열 시스템 (Queue Token)
  - 좌석 예약 도메인
  - Redis (분산락, 대기열 관리)
  - RDBMS (예약/좌석 데이터)

---

## 3. 부하 테스트 요약
### 3.1 테스트 대상
- 단일 콘서트 / 단일 날짜 / 단일 좌석에 대한 예약 API
- 가장 높은 경합과 장애 가능성이 존재하는 핵심 구간

### 3.2 테스트 도구
- k6 (ramping-vus 시나리오)

### 3.3 주요 결과
- 정상 응답: 409 (SEAT_ALREADY_RESERVED) 다수 발생
- 일정 트래픽 이상에서 500(SYS001) 비율 증가
- p95 응답시간 최대 3s 이상 증가

---

## 4. 장애 탐지 전략 (Detection Strategy)

### 4.1 장애 탐지 목적
- 사용자 인지 이전에 장애를 조기 감지
- 장애 확산 방지 및 빠른 대응
- 원인 분석을 위한 데이터 확보

---

### 4.2 장애 탐지 수단

#### 1) 부하 테스트 기반 사전 탐지
- k6 테스트 지표 활용
  - http_req_duration p95
  - http_req_failed rate
  - ErrorCode 발생 비율

**장애 판단 기준 예시**
- p95 응답시간 > 1.5s
- SYS001 비율 > 1%
- 예약 API 실패율 급증

---

#### 2) 애플리케이션 로그 기반 탐지
- ErrorCode 중심 로그 모니터링
  - SYS001 (INTERNAL_SERVER_ERROR)
  - 예상되지 않은 RuntimeException
- 동일 에러의 짧은 시간 내 반복 발생 시 장애로 분류

---

#### 3) 메트릭 기반 탐지 (운영 환경 가정)
- CPU / Memory 사용률
- JVM Heap 사용량
- Thread Pool 대기 큐
- Redis Lock 대기 시간
- DB 커넥션 풀 사용률

---

## 5. 장애 유형별 대응 시나리오

### 5.1 예측 가능한 장애

#### 유형
- 좌석 예약 트래픽 급증
- Redis 분산락 경합
- DB 커넥션 부족

#### 대응
1. 예약 API 임시 제한
2. 대기열 정책 강화
3. 서버 인스턴스 증설
4. 락 범위/시간 최적화 검토

---

### 5.2 예측 불가능한 장애

#### 유형
- 메모리 누수
- 처리되지 않은 Exception
- 외부 의존성 장애

#### 대응
1. 즉시 알림 확인
2. 장애 구간 격리
3. 서버 재시작 또는 롤백
4. 장애 원인 분석 및 재발 방지 조치

---

## 6. 장애 발생 시 대응 프로세스

1. 장애 탐지 (지표 / 로그 / 알림)
2. 장애 등급 분류
3. 담당자 알림
4. 즉각 대응 (차단 / 축소 / 롤백)
5. 정상화 확인
6. 사후 분석 및 문서화

---

## 7. 장애 회고 및 개선 프로세스

- 장애 타임라인 정리
- 사용자 영향 범위 분석
- 근본 원인 분석 (5 Whys)
- 단기/중기/장기 개선안 도출

---

## 8. 결론
본 장애 대응 매뉴얼은 부하 테스트 결과를 기반으로
실제 운영 환경에서 장애를 **사전에 감지하고**, **체계적으로 대응**하기 위한 기준을 제시한다.
이를 통해 서비스 안정성과 신뢰도를 지속적으로 개선할 수 있다.
