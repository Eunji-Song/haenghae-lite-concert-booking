# 좌석 예약 부하 테스트 계획서 (이선좌/동시성 충돌 시나리오)

## 1. 목적
콘서트 예매 오픈 시점에 특정 좌석으로 트래픽이 집중되면서 발생하는 **이선좌(이미 선택된 좌석) / 동시성 충돌** 상황을 테트기 위한 시리. 

- **정합성**: 동일 좌석에 대해 **중복 예약(HELD/CONFIRMED)이 발생하지 않을 것**
- **성능**: 피크(최고 부하) 구간에서도 API 응답 지연이 급격히 악화되지 않을 것
- **안정성**: 실패(이선좌 등) 응답이 폭주하더라도 서비스가 다운되지 않을 것(에러율, 타임아웃, DB 커넥션 고갈 관측)

> 이 과제 목적상 “이선좌”는 **Peak/Spike(최고 부하/스파이크)** 유형이 가장 잘 맞음  
> - 예매 오픈 순간 특정 좌석에 요청이 몰리는 “짧고 큰” 트래픽 패턴을 재현하기 좋음

---

## 2. 부하 테스트 대상 API 선정

### 2.1 대상 API
- **좌석 예약(홀드)**: `POST /api/v1/reservations`
  - Header: `Authorization: Bearer <accessToken>` (전역 보안 스키마)
  - Header: `X-Queue-Token: <queueToken>` (필수)
  - Body(JSON): `ReserveSeatRequest`
    - `concertId` (int64)
    - `date` (string)  예: `"2025-12-31"`
    - `seatNo` (int32)

### 2.2 왜 이 API인가?
- “이선좌”는 **동일 좌석**에 대한 동시 요청이 충돌하면서 발생하는 대표 케이스
- DB 트랜잭션/락 경합, 캐시/Redis 연동, 예약 상태 전이 등 **핵심 병목 지점**이 드러나기 쉬움
- 성공/실패 분포(성공은 소수, 실패는 다수)가 뚜렷하여 **에러 처리/리소스 사용량**을 관찰하기 좋음

---

## 3. 테스트 범위 및 제외 범위

### 3.1 포함
- API 레벨 부하: `POST /api/v1/reservations`
- 큐 토큰을 포함한 요청 흐름 (사전 발급/활성화)
- (선택) 예약 후 **결제 확정**까지 확장 시나리오
  - `POST /api/v1/payments` (Header: `X-Queue-Token`, Body: `PaymentRequest`)

### 3.2 제외(이번 과제에서는 최소화)
- 회원가입/로그인/토큰 재발급 성능 측정(사전 준비로 처리)
- 콘서트/좌석 조회 API 자체 성능 측정(필요 시 별도 시나리오로 확장)

---

## 4. 사전 준비(Precondition)

### 4.1 서버/인프라
- 서버: `http://localhost:8080`
- DB/Redis/Kafka 등 로컬 실행 상태 확인(이미 구성되어 있다면 그대로 사용)

### 4.2 테스트 데이터
- `concertId` 1개 고정(예: 1)
- 테스트 날짜 1개 고정(예: `"2025-12-31"`)
- **경합 유도 좌석 번호** 1개 고정(예: `seatNo=1`)
  - “이선좌”를 관찰하려면 **모든 가상 유저가 동일 좌석**을 누르도록 설계하는 게 핵심

### 4.3 인증/대기열 토큰 준비 (API Spec 기준)
1) (사전 1회) 로그인으로 `accessToken` 확보  
- `POST /api/v1/auth/login` → `UserToken.accessToken`

2) (시나리오 시작 전) 큐 토큰 발급  
- `POST /api/v1/queue/token`  
  - Body: `{ "concertId": 1 }`
  - Response: `queueToken`, `status`

3) (필요 시) 대기 상태 폴링하여 ACTIVE 진입 확인  
- `GET /api/v1/queue/me`  
  - Header: `X-Queue-Token: <queueToken>`
  - Response: `status`가 `ACTIVE`가 될 때까지 폴링

> 로컬 과제에서는 단순화를 위해  
> - ACTIVE가 바로 되는 환경이라면 폴링 단계를 생략해도 됨  
> - ACTIVE 전환 지연을 재현하려면 폴링을 시나리오에 포함

---

## 5. 테스트 시나리오

### 시나리오 A: 이선좌 스파이크(최고 부하) 테스트 (핵심)
- 목적: 오픈 순간 트래픽이 몰렸을 때 **중복 홀드 방지** 및 **시스템 안정성** 확인
- 요청: 모든 VU가 같은 `concertId/date/seatNo`로 예약 시도

**요청 예시**
- `POST /api/v1/reservations`
- Headers  
  - `Authorization: Bearer <accessToken>`
  - `X-Queue-Token: <queueToken>`
- Body
```json
{ "concertId": 1, "date": "2025-12-31", "seatNo": 1 }
```

**기대 결과**
- 성공(200) 응답은 극소수(대개 1명)  
- 나머지는 “이미 예약됨/홀드됨” 계열 예외(4xx)로 빠르게 반환
- 서버가 타임아웃/다운 없이 버텨야 함

---

### 시나리오 B: 피크 유지(스테디) + 좌석 분산 테스트 (선택)
- 목적: 특정 좌석이 아닌, 여러 좌석으로 분산 시 **처리량/응답시간** 측정
- VU마다 seatNo를 분산(예: 1~200 랜덤)하여 성공률과 평균 지연을 관찰

---

### 시나리오 C: 예약 → 결제 확정까지 E2E(선택)
- 목적: 예약 후 결제 단계(지갑 차감/예약 확정/이벤트 발행) 병목 확인
- 흐름:
  1) `POST /api/v1/reservations` 성공(홀드) 확보
  2) `POST /api/v1/payments`로 결제 & 예약 확정
- 이 시나리오는 “이선좌”보다는 **TPS/지연** 관점 확인에 더 유용

---

## 6. 부하 모델(부하 패턴) 제안

### 6.1 Spike(추천)
- Ramp-up: 5초 동안 VU 0 → N
- Peak 유지: 10~30초 유지
- Ramp-down: 5초
- N(동시 사용자)은 로컬 PC 성능에 따라 50/100/200 단계로 점진 증가

### 6.2 목표 지표(예시)
- p95 응답시간: 1s 이하 (로컬은 변동 가능, 상대 비교가 핵심)
- 에러율: 시나리오 A는 의도적으로 4xx가 많음  
  - “예상된 4xx(이선좌)”는 정상  
  - 5xx/타임아웃이 급증하면 문제

---

## 7. 관측/수집 항목 (최소)
- 애플리케이션 로그: 예약 성공/실패 사유, 예외 스택, 응답시간 로그(있다면)
- DB: 커넥션 수, 락 대기/데드락 여부, 슬로우 쿼리
- Redis(사용 시): ops/latency
- JVM/컨테이너 리소스: CPU, 메모리, GC, 스레드 수

> 과제 요구사항에 “도커 CPU/Memory 제한을 조절하며 적정 스펙 탐색”이 포함될 수 있으니  
> 컨테이너 리소스를 줄이면서 p95/에러율이 어떻게 변하는지 기록하면 좋음.
