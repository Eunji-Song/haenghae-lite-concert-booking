# 🎫 콘서트 예약 서비스 MSA 전환을 위한 Transaction Diagnosis 설계 보고서

본 문서는 기존 모놀리식 형태의 콘서트 예약 서비스를  
**MSA(Microservice Architecture)** 구조로 확장할 경우를 가정하여,  
도메인 분리 기준과 그에 따른 **트랜잭션 처리 한계**, 해결하기 위한 **설계 전략**을 정리한 문서

---

## 1. 문제 배경

콘서트 예약 서비스는 다음과 같은 특성을 가진다.

- 좌석 수는 한정되어 있으며, 동시성 이슈가 빈번하다
- 결제, 좌석 점유, 잔액 차감 등 다수의 상태 변경이 연쇄적으로 발생한다
- 트래픽이 특정 시점(오픈 시각)에 집중된다

서비스 규모가 확장되면 단일 애플리케이션 및 단일 DB 구조는 다음 한계를 가진다.

- 트랜잭션 범위가 과도하게 커짐
- 장애 발생 시 전체 서비스 영향
- 도메인별 배포 및 확장 불가

이에 따라 도메인 단위로 배포 가능한 **MSA 구조 전환**이 필요.

---

## 2. 도메인 분리 기준

| 도메인 | 책임 |
|---|---|
| Identity | 사용자 인증, JWT 발급 |
| Queue | 대기열 토큰 발급 및 상태 관리 |
| Product | 콘서트, 날짜, 좌석 정보 |
| Reservation | 좌석 임시 홀드 / 확정 |
| Wallet | 사용자 잔액 조회 및 충전 |
| Payment | 결제 처리 |
| Event | 도메인 간 상태 전달 |

---

## 3. 트랜잭션 처리의 한계

MSA 구조에서는 각 도메인이 서로 다른 DB를 사용하므로  
단일 트랜잭션으로 모든 작업을 묶는 것이 불가능하다.

이에 따라 **최종 일관성(Eventual Consistency)** 을 전제로 한 설계가 필요하다.

---

## 4. 해결 전략

- 도메인 내부: 로컬 트랜잭션 보장
- 도메인 간: 이벤트 기반 비동기 처리
- 중복 요청 방지: 멱등성 키 사용

---

## 5. 이벤트 기반 처리 흐름

결제 완료 시 `PaymentCompletedEvent`를 발행하고  
각 도메인은 이를 수신하여 자신의 책임 범위 내 작업을 수행한다.

---

## 6. 결론

본 설계는 확장성과 장애 격리를 고려한 현실적인 MSA 전환 전략이며,  
콘서트 예약과 같은 고동시성 서비스에 적합한 구조를 제공한다.
